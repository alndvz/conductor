What Is This?

Conductor Agent is an OpenCode plugin that watches TASKS.md, delegates to specialized AI sub-agents, reviews their output, and commits — all without human intervention.

Think: AI project manager + coding team + QA, working asynchronously around the clock. The Conductor orchestrates, the Implementor builds, the Reviewers inspect — and the Conductor arbitrates their feedback intelligently before committing.

Conductor

primary agent

Orchestrator, arbiter, committer. Reads TASKS.md, delegates to sub-agents, evaluates review feedback with intelligent arbitration, and commits approved changes.

Model: deepseek-v4-pro
Only agent with permission: allow

Feature

primary agent

Conversational-first planner. Discusses requirements with the user, asks clarifying questions, writes TASKS.md tasks and conductor-plans. Can delegate to the explore sub-agent for codebase exploration.

Model: — (default)

Implementor

sub-agent

Writes code. Has read, write, edit, bash, glob, and grep tools. Stateless — each invocation runs in a fresh context. Reports completion to the Conductor.

Model: — (default)

Review

sub-agent

General code review. Read-only tools (Read, Bash, Glob, Grep). Returns APPROVED or CHANGES REQUESTED with specific, actionable feedback.

Model: deepseek-v4-pro

Rules Review

sub-agent

Domain rule checker. Evaluates 2 rules (simplicity, minimal-diff) against git diff. Reports PASS/FAIL per rule. Read-only tools.

Model: deepseek-v4-flash

How It Works

User writes a feature request or task

The process starts with a human request — a feature idea, a bug report, or any task that needs code changes.

Feature agent discusses and captures

Feature agent converses with the user, asks clarifying questions, and captures the result into TASKS.md and/or conductor-plans/.

Plugin detects the change

The plugin polls the git hash every 5 seconds. When TASKS.md changes, it computes the diff and notifies the Conductor.

Conductor receives the notification

The plugin sends the diff to all active Conductor sessions. The Conductor reads the updated tasks and begins orchestrating.

Conductor delegates to Implementor(s)

Each incomplete task is delegated to an Implementor sub-agent with clear instructions and acceptance criteria.

Plan-based tasks: parallel fan-out

For conductor-plans/ tasks, independent steps run in parallel (multiple Implementors at once), while dependent steps run sequentially.

Implementor writes code and reports back

The Implementor uses its tools to write, edit, and verify the changes, then reports completion to the Conductor.

Dual-layer review launched

The Conductor launches both Review (general code review) and Rules Review (domain rule check) in parallel against the combined diff.

Conductor arbitrates feedback

If either review returns CHANGES REQUESTED, the Conductor evaluates the feedback: legitimate → send back to Implementor; nonsensical → push back on reviewer; cyclical → escalate to user; minor → override and commit.

Conductor commits and marks as done

With all reviews passed (or overridden), the Conductor commits with a descriptive message and marks the task [x] in TASKS.md.

Key Features

Plugin-Based Orchestration

Guard-based activation (CONDUCTOR=1), session management, message queueing with backpressure, defensive idle detection, and self-contained cleanup.

Plan-Driven Parallelism

Multi-step plans fan out independent steps to parallel Implementors, then run both reviews simultaneously on the combined diff.

Dual-Layer Review

General code review and rules-based review run concurrently — two specialized agents examining the same diff from different angles.

Intelligent Arbitration

The Conductor doesn't blindly follow reviewers. It evaluates feedback, pushes back when appropriate, and escalates only when truly needed.

Self-Installing

/install-conductor bootstraps the entire Conductor workflow into any target repo — agents, plugins, config, and scaffolding.

Containerized Dev

dev.sh builds and runs a Fedora Podman container with the full development environment, so you can hack on the Conductor itself.

Architecture Documentation

/detect-arch command auto-generates architecture docs analyzing the codebase structure, patterns, and relationships.

File Change Detection

Git hash polling with diff computation at 5-second intervals — robust against clock skew and designed for the event-driven agent loop.

Architecture Deep Dive

Plugin Design

  • Activation gate: only runs when CONDUCTOR=1 env var is set
  • Polling loop: git hash every 5s, diffs on change, notifies sessions
  • Session tracking: session.created / session.deleted events
  • Message queueing with backpressure: queues outbound messages when sessions are busy, drains on idle
  • Defensive idle detection: avoids stale session states
  • Fire-and-forget messaging: non-blocking notification dispatch
  • Self-contained cleanup: no leaked timers or listeners

Plan Execution Model

  • Plans in conductor-plans/ have a ## What section with numbered steps
  • Independent steps → launched as parallel Implementor sub-agents in a single message
  • Dependent steps → run sequentially, waiting for prior output
  • After all Implementors finish → both reviewers run once on the combined diff
  • Both reviews must pass before committing

Arbitration Logic

  • Legitimate: specific, actionable, non-contradictory → forward to Implementor for fixes
  • Nonsensical: misunderstands task, contradicts spec, misapplies rule → push back on reviewer
  • Cyclical: same issue returns after 2+ rounds → escalate to user
  • Minor: cosmetic, subjective, outside scope → override and commit
  • Override recorded in commit message: (review overridden: reason)
